# Glossary

| Keyword | Plain-English meaning | What it does | Where it appears in Databricks | Why it matters | Common confusion | Related terms |
|---|---|---|---|---|---|---|
| Delta Lake | Lakehouse table format with transaction log | Adds ACID and versioning over Parquet | Default table format across workspace | Reliable batch + streaming + updates | "Just Parquet" misconception | transaction log, ACID, CDF |
| transaction log | Delta metadata history (`_delta_log`) | Tracks commits and table versions | Delta tables | Enables time travel and consistency | Editing files directly bypasses it | table history, ACID |
| ACID transactions | Atomic/consistent/isolation/durability guarantees | Protects correctness under concurrent writes | Delta operations | Prevents partial/corrupt updates | Assuming eventual consistency is enough | optimistic concurrency |
| schema enforcement | Write-time schema validation | Blocks incompatible writes | Delta tables | Keeps downstream pipelines stable | Confused with schema evolution | constraints, schema evolution |
| Structured Streaming | Spark stream processing engine | Continuous incremental processing | Notebooks, jobs, pipelines | Real-time/near-real-time data flows | Confused with Auto Loader role | Auto Loader, checkpoints |
| Lakeflow | Databricks pipeline suite | Managed ingestion/orchestration options | Lakeflow docs and UI | Production-grade pipeline operations | Treated as one single service only | Lakeflow SDP, Jobs, Connect |
| Spark Declarative Pipelines | Framework for SQL/Python data pipelines | Defines streaming tables/materialized views | `ldp` pipelines | Simplifies incremental ETL | Old DLT naming overlap in memory | Auto Loader, streaming tables |
| RAG | Retrieval-augmented generation pattern | Grounds LLM answers on external context | GenAI apps, agents, notebooks | Reduces hallucination and adds freshness | Mistaken as only vector search | retrieval, augmentation, generation |
| retrieval | Fetching supporting context for a query | Gets relevant chunks/rows/docs | Retriever/vector search/query logic | First quality gate in RAG | Confused with generation quality | retriever, context precision |
| augmentation | Building model prompt with retrieved context | Injects evidence into prompt | RAG chain logic | Improves grounded answers | Treated as same step as retrieval | prompt template, context |
| generation | LLM response creation step | Produces answer from prompt/context | Model Serving/LLM endpoint | Final user-visible output | Blamed for retrieval failures | LLM, groundedness |
| chunking | Splitting docs into smaller retrieval units | Balances context and granularity | Unstructured pipeline steps | Major driver of retrieval quality | Bigger chunks assumed always better | overlap, embeddings |
| embeddings | Vector representations of text/images | Encodes semantic meaning for search | Foundation model APIs, Vector Search | Enables semantic retrieval | Vector = keyword score confusion | vector index, similarity |
| vector index | Search structure over embeddings | Fast ANN/hybrid retrieval | Mosaic AI Vector Search | Core RAG retrieval infrastructure | Same as source Delta table assumption | endpoint, ANN, HNSW |
| Vector Search | Databricks managed vector retrieval service | Hosts endpoints/indexes over Delta data | Mosaic AI Vector Search | Governed scalable semantic search | Endpoint type implications overlooked | hybrid search, reranking |
| retriever | Component selecting candidate context | Executes search/filter/top-k logic | Agent/RAG chain code | Determines evidence quality | Retriever vs reranker confusion | retrieval, reranking |
| reranking | Second-pass candidate ordering | Improves relevance of top context | Vector Search query options/chain logic | Boosts quality on ambiguous queries | Free quality gain assumption | latency trade-off, top-k |
| ai_parse_document | AI Function for document parsing | Extracts text/layout/tables from docs | AI Functions (Public Preview) | Speeds document ingestion for RAG | Parsed output quality assumed perfect | ai_prep_search, chunking |
| Agent Bricks | Managed experiences for GenAI agents | Builds agents with Databricks controls | Agents section/docs | Faster path from prototype to deployment | Confused with custom DIY framework | Knowledge Assistant |
| Knowledge Assistant | Managed document Q&A agent | Builds citation chatbot over docs/indexes | Agent Bricks | Rapid RAG deployment + feedback loops | Source sync requirements forgotten | vector index, citations |
| MLflow | Experiment + model/app lifecycle platform | Tracks runs, evaluates, manages registry | Databricks MLflow integration | Reproducibility and controlled iteration | OSS vs Databricks-managed feature mix | logged models, registry |
| Model Serving | Managed endpoint hosting for models/agents | Exposes REST inference endpoints | Mosaic AI Model Serving | Production inference at scale | Serving endpoint vs registry confusion | inference tables, AI Gateway |
| inference tables | Logged request/response payload tables | Capture serving traffic for monitoring | Model Serving / AI Gateway | Debugging, drift checks, feedback datasets | Legacy vs AI Gateway variants | monitoring, payload schema |
| Unity Catalog | Unified governance layer for data + AI assets | Central permissions, lineage, auditing | Account/workspace governance | Mandatory governance foundation | Workspace-only security assumptions | metastore, privileges |
| model lineage | Trace from model to data/runs | Shows upstream dependencies | UC + MLflow model pages | Compliance and root-cause analysis | Expecting lineage without logging inputs | mlflow.log_input, catalog lineage |
| model registry | Managed model version repository | Stores versions/aliases/tags | MLflow + Unity Catalog | Controlled promotion and rollback | Version numbers hardcoded in apps | aliases, Champion |
| governance | Policies + controls over data/AI lifecycle | Enforces access, audit, quality guardrails | Unity Catalog and related tooling | Required for enterprise/regulated use | Seen as only permissions | auditing, lineage |
| permissions | Explicit privileges for securables | Controls who can read/write/execute | UC GRANT model, endpoint ACLs | Prevents unauthorized access | Missing `USE CATALOG/SCHEMA` prerequisites | GRANT, EXECUTE |
| evaluation metrics | Quantitative/LLM-judge quality signals | Measure retrieval, response, and ops | MLflow eval/monitoring | Guides iterative improvement | Mixing retrieval and response metrics | correctness, faithfulness |
| faithfulness | Output consistency with provided context | Detects context-grounding quality | DeepEval scorer in MLflow integration | Finds hallucination risk with context | Same as correctness assumption | groundedness, context |
| answer correctness | Match to expected facts/ground truth | Checks factual correctness vs truth | Correctness judge / DeepEval alternatives | Strong offline quality benchmark | Needs ground truth, unlike relevance | expected_response, expected_facts |
| answer relevancy | Response relevance to user query | Scores topical alignment of answer | DeepEval `AnswerRelevancy` | Detects off-topic responses | Can be high even if factually wrong | relevance_to_query |
| context precision | Relevance of retrieved nodes ranking | Measures noisy vs useful retrieval set | DeepEval `ContextualPrecision` | Diagnoses retriever ranking quality | Confused with context recall | retrieval relevance, top-k |
| context recall | Coverage of necessary supporting info | Measures if retrieval missed needed info | DeepEval `ContextualRecall` | Diagnoses missing context failures | Requires notion of needed information | document recall, sufficiency |
